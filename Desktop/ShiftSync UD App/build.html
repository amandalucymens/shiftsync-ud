Build: generate your ShiftSync UD app

This is the exciting part: you’ll paste one “build prompt” into Cursor, and it will generate a full Next.js app structure for ShiftSync UD—pages, database setup, login, and the core flows.

Mental model: you’re the product manager

You’re not expected to write code from scratch. Your job is to describe what ShiftSync UD should do (like a product spec). Cursor will act like an engineering team and produce files. You then test the app and ask Cursor to fix issues.

  1. Describe what you want (prompt).
  2. Run the app locally.
  3. Observe what’s working / what’s broken.
  4. Paste errors into Cursor and ask it to fix them.

ShiftSync UD Build Prompt (copy/paste)

Tip: this is intentionally detailed. The goal is to reduce back-and-forth and get a working first version fast. 

Build me a Next.js web application. Here are the complete specifications:

## Tech Stack (DO NOT DEVIATE)
- Next.js 14+ with App Router (not Pages Router)
- TypeScript
- Tailwind CSS for styling
- InstantDB for database and authentication (use @instantdb/react)

## InstantDB Configuration
My InstantDB App ID is: YOUR_APP_ID_HERE

Initialize InstantDB in a file at `src/lib/db.ts` like this:
```typescript
import { init } from '@instantdb/react';
import schema from '../../instant.schema';

const APP_ID = 'YOUR_APP_ID_HERE';

export const db = init({
  appId: APP_ID,
  schema
});
```

## App Description
ShiftSync UD is a shift-management app for UD Catering student servers and student managers. Servers create a profile (including uniform size and manager notes), submit weekly availability in a grid, and view their assigned shifts. When a server can’t make a shift, they post a swap request; eligible servers can claim it; managers approve or deny the swap and the published schedule updates automatically—no email threads.

## Data Model
Create an InstantDB schema in `instant.schema.ts` with these entities (keep it beginner-friendly and within 2–4 core entities; use links for relationships):

- `profiles`
  - displayName (string)
  - contact (string) // phone or email to help managers reach them
  - uniformSize (string) // one of XS, S, M, L, XL, XXL
  - notes (string) // freeform info managers should know
  - role (string) // "server" | "manager"

- `availabilityBlocks`
  - dayOfWeek (string) // "Mon"..."Sun"
  - timeBlock (string) // "morning" | "afternoon" | "evening"
  - isAvailable (boolean)

- `shifts`
  - eventName (string)
  - date (date)
  - timeBlock (string) // "morning" | "afternoon" | "evening"
  - location (string)
  - meetupDetails (string)
  - status (string) // "draft" | "published"

- `swapRequests`
  - status (string) // "open" | "claimed" | "approved" | "denied"
  - createdAt (date)

Relationships (InstantDB links):
- Each `profile` belongs to exactly one `$users` user.
- Each `availabilityBlock` belongs to exactly one `profile`.
- Each `shift` can have many assigned `profiles` (start simple: assign 1 server per shift in V1; structure code so it can be extended to multiple later).
- Each `swapRequest` belongs to exactly one `shift` and has:
  - `fromProfile` (the person dropping the shift)
  - `toProfile` (optional; the person claiming it)

## Permissions (InstantDB)
Create `instant.perms.ts` with permissions that are correct and safe, but not overly complex for a first app:
- Only authenticated users can read/write data.
- A user can create and update their own `profile`.
- A user can create and update their own `availabilityBlocks`.
- Servers can read `shifts` that are assigned to them.
- Servers can create a `swapRequest` only for a `shift` assigned to them.
- Servers can read open `swapRequests` (we will filter to “relevant to me” in the UI for V1).
- Managers (role="manager") can read all `profiles`, `availabilityBlocks`, `shifts`, and `swapRequests`, and can approve/deny swap requests.

If something is not possible to enforce perfectly in perms without lots of complexity, enforce it in the UI for V1 and explain the tradeoff in code only when necessary.

## Authentication
Implement user authentication using InstantDB's Magic Code flow:
1. Use `db.useAuth()` to check login status
2. User enters email
3. InstantDB sends a 6-digit magic code
4. User enters code to finish login
5. Protect routes that require authentication (redirect to /login)

## Pages to Create (App Router)
- `/` Home dashboard (requires auth): role-aware landing page
  - If server: quick links to Profile, Availability, My Schedule, Swap Board
  - If manager: quick links to Manager Schedule Builder + Approvals
- `/login` Login page with Magic Code flow
- `/profile` Profile create/edit page (requires auth)
- `/availability` Weekly availability grid page (requires auth)
- `/schedule` Server page: "My Schedule + Swap Board" (requires auth)
  - Show assigned shifts
  - Allow posting a swap request for a shift
  - Show open swap requests that match my availability (client-side filtering)
  - Allow claiming a swap request
- `/manager` Manager Schedule Builder (requires auth, manager-only)
  - Create shifts/events for the week (eventName, date, timeBlock, location, meetupDetails)
  - Assign a server to a shift (choose from profiles; show uniform size + notes for convenience)
  - Publish shifts (set status="published")
  - Show pending swap requests and allow approve/deny

## Core Features (MVP)
1. Magic Code login (no passwords)
2. Profile creation with uniform size + manager notes
3. Availability grid that saves to InstantDB
4. Server schedule view (assigned shifts)
5. Swap board: post swap requests, claim swaps, manager approve/deny, schedule updates
6. Manager schedule builder: create shifts and assign servers

## UI Requirements
- Clean, modern Tailwind UI (mobile responsive)
- Clear navigation between pages (top nav or sidebar)
- Loading states while queries load
- Error messages when something fails
- Use reusable components where sensible (e.g., Layout/Nav, Button, Card)

## Getting Started Steps (must include)
1. Create the Next.js app in the current folder:
   `npx create-next-app@latest . --tailwind --eslint --app --src-dir --import-alias \"@/*\" --typescript`
2. Install InstantDB:
   `npm install @instantdb/react`
3. Create the schema and permissions files and push them using InstantDB CLI commands that are appropriate for InstantDB.
4. Ensure the app runs locally: `npm run dev`

Please create all necessary files with complete, working code and no placeholders.

3) Run your app locally

Mental model: running locally is like a “practice kitchen” on your own laptop. It’s not on the internet yet—only your computer can see it.

  1. In Cursor, open the terminal (View → Terminal).
  2. Run:
npm run dev

Then open http://localhost:3000.

What is “localhost”?

localhost is a special address that means “this computer.” The :3000 part is like a specific door number. Your app is temporarily using door 3000 on your laptop.

4) What to expect (and how to handle issues)

It’s normal for the first run to have a few bumps. The skill you’re building is not “memorize syntax”—it’s learning how to iterate: run → see error → ask Cursor to fix → repeat.

If something breaks, do this

  1. Copy the full error message (from the terminal or browser).
  2. Paste it into Cursor and say: “I’m getting this error. Please fix it.”
  3. Let Cursor apply the fix, then run npm run dev again.
