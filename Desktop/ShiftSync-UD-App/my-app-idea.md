# ShiftSync UD

## One-Sentence Summary

ShiftSync UD is a shift-management app for UD Catering student servers and managers to organize availability, assign shifts, and handle swap requests — without the email chaos.

## The Problem

UD Catering managers and student servers currently coordinate entirely through emails and a shared Google Doc. Servers email their availability by hand. Managers manually match each event to available servers. Every Thursday, a Google Doc goes out with the week's assignments — and when someone needs to swap a shift, it triggers a flood of reply-all emails to the entire server pool, cluttering inboxes with messages that don't apply to most people.

There's also no central place to store server information. For off-campus events, managers need each server's uniform size in advance — but that detail lives in scattered emails or nowhere at all.

ShiftSync UD replaces this with one platform where server profiles store everything managers need, availability is always current, and swap requests only go to the people they're relevant to.

## User Types

- **Student Servers** — Create a profile, submit weekly availability, view assigned shifts, and post or claim shift swaps.
- **Student Managers** — View all server profiles and availability, create and publish shift assignments, and approve swap requests.

## Core Features

- Create a server profile with uniform size and logistical notes managers need to know
- Submit and update weekly availability on a simple grid — no email required
- View your assigned shifts for the week, including event location and meetup details
- Post a swap request when you can't make a shift, or claim an open swap from another server
- Approve or deny swap requests and keep the published schedule up to date (managers only)

## What Users See

### Screen 1: Server Profile
A form where servers enter their name, contact info, uniform size (XS–XXL), and a freeform notes field for anything managers should know — like "commutes from off campus, needs 30 min notice for location changes." Managers can pull up any server's profile card directly from the schedule builder.

### Screen 2: Availability Settings
A weekly grid where servers toggle their availability by day and time block. Saves automatically and is always visible to managers — no follow-up needed.

### Screen 3: My Schedule + Swap Board
Servers see their upcoming assigned shifts in one place: event name, date, time, and meetup location. From this same screen, they can post a swap request on any shift they need to drop. Open swap requests are only visible to servers who are available during that time slot — not the whole server pool. A server who can cover it claims the request, and the manager receives a notification to approve or deny the change.

### Screen 4: Manager Schedule Builder
A weekly view of all upcoming events. For each event, managers see a filtered list of available servers with profile details — uniform size and notes — visible at a glance. Managers assign servers and publish the schedule. Pending swap requests appear here for manager review, and approving one automatically updates the published schedule.

## Example Scenario

Maya is a sophomore server at UD Catering. She fills out her ShiftSync UD profile — medium uniform, and a note that she commutes from Newark and needs advance notice for off-campus meetups.

On Sunday, she updates her availability for the week: free Monday, Wednesday, and Friday evenings.

On Monday, her manager Jordan opens the schedule builder, sees a fundraising dinner on Wednesday at 6 PM, and pulls up the list of available servers. Maya's profile card shows her uniform size and commute note right there — Jordan pulls a medium uniform from the storeroom and assigns Maya and three other servers. Maya gets a notification with the event details and meetup time.

On Tuesday, Maya realizes she has a midterm conflict. She opens her schedule, posts a swap request for the Wednesday shift, and it appears on the Swap Board — but only for servers who are free Wednesday evening. Another server, Dev, sees the open request and claims it. Jordan gets a notification, reviews it, and approves the swap. The schedule updates automatically, and both Maya and Dev are notified.

No reply-all emails. No scrambling. Jordan already has Dev's uniform size on file.
