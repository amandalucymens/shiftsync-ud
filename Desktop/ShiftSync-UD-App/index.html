What you’re building: ShiftSync UD

ShiftSync UD is a shift-management web app for UD Catering student servers and student managers. Servers create a profile (including uniform size + notes), set weekly availability in a grid, and see their assigned shifts. If a server can’t make a shift, they post a swap request; another eligible server can claim it, and a manager approves the change—without reply-all email chaos.

Student servers can…

  • Create and update their profile (uniform size, contact, notes)
  • Submit weekly availability on a simple grid
  • View assigned shifts (event name, time, meetup location)
  • Post or claim shift swaps

Student managers can…

  • See all server profiles and availability
  • Create and publish shift assignments
  • Review and approve/deny swap requests
  • Keep the schedule up to date automatically

Mental model: website vs web app

A website is usually like a brochure: the pages are mostly the same for everyone (think: a restaurant menu page). A web app is like a tool: it can have accounts, saved data, and personalized screens (think: Google Docs).

ShiftSync UD is a web app because it needs:

  • Accounts (each server and manager logs in)
  • Saved data (availability, profiles, schedules)
  • Multiple users seeing shared reality (the same schedule)

Mental model: the client–server model (restaurant analogy)

Here’s the simplest useful way to understand how most web apps work. Imagine a restaurant:

Frontend (Client)

The dining room. What users see: buttons, forms, pages in the browser.

API (Requests)

The waiter. Carries messages between the dining room and the kitchen.

Backend (Server + DB)

The kitchen + pantry. Stores data, checks permissions, returns results.

When a server updates availability, your frontend sends a request (like placing an order). The backend stores it so managers can see it too (like the kitchen preparing food for the whole restaurant).

Why you need both

  • Security: you can’t trust data that only lives in someone’s browser.
  • Shared truth: Maya and Jordan need to see the same schedule.
  • Persistence: if you close the laptop, the schedule can’t disappear.

The tools you’ll use (and why)

You’re going to use a modern “starter stack” that’s beginner-friendly and deploys cleanly to the internet.

Tool What it does Analogy
Cursor AI-powered code editor that helps generate and fix code. Your pair-programmer
Next.js Framework for building the UI and pages (React + routing). Blueprint for the dining room
InstantDB Hosted backend: database + authentication. Kitchen + pantry, fully staffed
Vercel Hosting that puts your app on the internet. The building that houses your restaurant
GitHub Stores your code and version history, connects to Vercel. A filing cabinet for your blueprints
